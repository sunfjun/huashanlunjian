# Proposal: Agora Token 优化方案
**Created:** 20260325-120000
**Roles:** 🔮Cassandra 🦉Athena 🎭Momus ⚒️Hephaestus
**Discussion rounds:** 4
**Status:** Consensus reached

---

## 背景与目标

Agora 当前的 token 消耗主要来自：每轮 4 个 opus agent 调用（各接收完整提案 + 上轮上下文 + 角色 prompt + review 指令），提案全文重复发送，上轮角色完整输出回传，orchestrator prompt 约 300 行。

**目标：在不影响讨论质量的前提下，最大化 token 节省。**

---

## 优化方案

### 优化 1：智能角色休眠（节省 ~15-25% agent 调用）

**现状：** 每轮固定调用所有 4 个角色。

**优化：** 纯机械规则，无需 LLM 判断。

**触发条件：**
- 某角色连续 2 轮 "No objection" **且** `proposal_modified == false`
- `proposal_modified` 是二元标志，由 orchestrator 在每轮结束时设置：accept 了任何 objection 并修改了提案文本 → `true`，否则 → `false`

**安全机制：**
- 强制唤醒：每 3 轮强制调用一次休眠角色
- 唤醒时始终发送完整提案（非 diff）
- 第一轮始终调用所有角色
- 休眠角色唤醒时接收错过轮次的累积 Key Points（按轮次列出）
- Sleep/wake 状态在任何歧义时默认为 "awake"

### 优化 2：压缩 agent prompt 上下文（节省 ~20-30% input token/agent）

**现状：** 每个 agent 收到完整提案 + 完整上轮输出 + 所有角色摘要。

**优化：** 三个子项，均保留完整提案传递。

#### 2a. 结构化前轮状态替代完整输出（正式规范）

替换 "This Role's Previous Round Full Output" 为以下格式：

```
你上轮提出的问题及处理状态：
1. [你的原始 objection 原文] → 状态: accepted
   Orchestrator 回应: [1-2句摘要]
2. [你的原始 objection 原文] → 状态: rejected
   Orchestrator 回应: [拒绝理由摘要]
3. [你的原始 objection 原文] → 状态: ongoing
   Orchestrator 回应: [当前进展摘要]
   行动指引: 请重新评估此问题在当前提案版本中是否仍然存在，并明确标注为 resolved 或 still unresolved，附上更新后的理由。

你上轮的 Summary: [原文保留]
```

**设计要点：**
- 包含角色原始 objection 原文（Issue 行），允许角色独立验证 orchestrator 的处理是否准确
- 为 "ongoing" 状态提供明确的行动指引，避免不同角色对 "ongoing" 的理解不一致
- "Ongoing" 行动指引应机械生成（非 LLM 总结），避免增加额外 token 成本
- 比完整输出短约 50-60%

#### 2b. 静态格式指令移入 agent 文件

- 将 Review Instructions 中的固定部分（格式要求、Key Rules）写入各角色 .md 文件
- prompt 中只保留动态内容：当前提案、上轮状态、unresolved issues

#### 2c. 精简跨角色信息

- 其他角色的 Key Points 保留（有效的跨角色信息共享）
- 移除其他角色的 Summary（与 Key Points 信息重复）

### 优化 3：收敛加速（减少 1-2 轮）

**现状：** 需要连续 2 轮所有角色 No objection 才收敛。

**优化：**
- 保持 2 轮收敛作为**默认规则**
- **快速收敛条件**（全部必须满足）：
  1. `proposal_modified == false` 连续 2+ 轮
  2. 最近一轮所有**活跃**角色 No objection
  3. 所有角色（包括休眠角色）的 `last_reviewed_version >= current_proposal_version`（即所有角色都曾审阅过当前提案版本）
  - 若条件 3 不满足：先唤醒未审阅当前版本的角色，等待其响应后再判断
- 重复 objection 标记 "Already addressed in Round N"，不重复传递

### 优化 4：prompt 模板精简（节省 ~10-15% orchestrator token）

- 将 Anti-Patterns 表移至独立文档（开发参考，按需加载）
- 将 Pre-Release Checklist 移至独立文档
- 精简重复格式说明
- 目标：proposal.md 从 ~300 行压缩到 ~200 行
- Anti-Patterns/Checklist 文档应按需加载（仅在角色活跃时），而非始终捆绑

## 实施步骤

### Phase A（立即实施）：优化 2 + 优化 4

纯 prompt 文本重组，无行为逻辑变更。

**具体操作：**
1. 修改各角色 .md 文件，加入 Review Instructions 固定部分
2. 修改 proposal.md orchestrator prompt，移除冗余格式指令
3. 实现结构化前轮状态模板
4. 将 Anti-Patterns 表和 Pre-Release Checklist 拆分为独立文件

**成功标准：** 每个 agent 的 input token 减少 ≥20%，讨论轮次不增加超过 1 轮

### Phase B（Phase A 验证后）：优化 1 + 优化 3

引入状态跟踪和行为逻辑变更。建议 Phase A 与 Phase B 之间留出 bake period 观察回归。

**具体操作：**
1. 在 orchestrator prompt 中添加休眠逻辑（proposal_modified 标志、连续 No objection 计数、强制唤醒计时器）
2. 添加 last_reviewed_version 跟踪
3. 实现快速收敛的三重条件检查
4. 实现累积 Key Points 传递

**成功标准：** 总 agent 调用次数减少 ≥15%，无误判休眠案例

## 基线测量计划

在实施前，使用当前系统运行 3 个代表性讨论，记录每个角色每轮的精确 token 数：
- 1 个高争议讨论（预期 4+ 轮，多 objection）
- 1 个中等讨论
- 1 个顺畅讨论（预期 2-3 轮）

## 未来探索（当前不实施）

- **模型降级策略：** 需要对比测试确认 Sonnet review 质量。如果实施，Sonnet 的 "No objection" 不计入收敛阈值。需要定期 Opus spot-check 防止单向棘轮效应。
- **提案 diff 传递：** 仅在有可靠的 diff 机制且有状态 agent 支持时考虑。当前无状态 agent 架构不适合 diff。
