## Round 1

### 💡 Claude:

**背景与目标**

Agora 当前的 token 消耗主要来自以下几个方面：

1. **每轮 4 个 opus agent 调用** — 这是最大的成本来源。每个 agent 接收完整提案 + 上轮上下文 + 角色 prompt + review 指令
2. **提案全文在每轮重复发送** — 每轮的 "Updated proposal" 全文嵌入每个 agent 的 prompt 中
3. **上轮角色完整输出回传** — "This Role's Previous Round Full Output" 将该角色上轮的完整回复发回
4. **orchestrator prompt 本身约 300 行** — 每次会话都完整加载
5. **讨论文件 append-only** — 虽然不直接影响 agent token，但影响主 Claude 的上下文

**目标：在不影响讨论质量的前提下，最大化 token 节省。**

---

**提案设计**

### 优化 1：智能角色选择（节省 ~25-50% agent 调用）

**现状：** 每轮固定调用所有 4 个角色。
**优化：**
- 当某角色连续 2 轮 "No objection" 且无新提案变更涉及其关注领域时，该角色进入"观察模式"，跳过该轮调用
- 观察模式的角色在有新的重大提案变更时自动唤醒
- 第一轮始终调用所有角色

### 优化 2：压缩 agent prompt 上下文（节省 ~30-40% input token/agent）

**现状：** 每个 agent 收到完整提案 + 完整上轮输出 + 所有角色摘要。
**优化：**
- **提案差异化传递**：Round 1 发送完整提案，后续轮次只发送提案的变更部分（diff）+ 不变部分的摘要
- **移除 "This Role's Previous Round Full Output"**：角色已有 Key Points 和 Summary 机制，完整回传是冗余的。用该角色的 Key Points + Summary 替代
- **精简 Review Instructions**：将固定的格式指令移到角色 agent 文件中（一次性），prompt 中只保留当前轮次的动态内容

### 优化 3：角色 agent 降级策略（节省 ~60-70% 特定调用成本）

**现状：** 所有角色始终使用 opus 模型。
**优化：**
- 当角色上轮 "No objection" 且本轮提案变更较小时，该角色的 agent 调用降级为 sonnet
- 仅在角色切换到 "Objection" 或提案有重大变更时升回 opus
- 这样"确认性"review 用更便宜的模型，"批判性"review 保持 opus 质量

### 优化 4：收敛加速（减少总轮次）

**现状：** 需要连续 2 轮所有角色 No objection 才收敛。
**优化：**
- 引入"实质性变更"检测：如果本轮提案与上轮完全一致（无 accept 任何新 objection），且所有角色 No objection，则 1 轮即可收敛
- 对于重复的 objection，orchestrator 直接标记 "Already addressed" 而不传递给其他角色

### 优化 5：prompt 模板精简（节省 ~10-15% orchestrator token）

**现状：** proposal.md 约 300 行，包含大量示例和反模式表。
**优化：**
- 将 Anti-Patterns 表和 Pre-Release Checklist 从 prompt 中移除（这些是开发指南，不需要每次运行时加载）
- 精简重复的格式说明
- 预计可将 proposal.md 从 ~300 行压缩到 ~180 行

---

**实施步骤**

1. **Phase A（低风险，高收益）**：实施优化 2（压缩 prompt）和优化 5（模板精简）
2. **Phase B（中等风险，高收益）**：实施优化 1（智能角色选择）和优化 4（收敛加速）
3. **Phase C（需验证）**：实施优化 3（模型降级），需要对比测试确认 sonnet 的 review 质量

---

**预期效果**

| 优化项 | Token 节省估计 | 风险等级 |
|--------|---------------|---------|
| 智能角色选择 | 25-50% agent 调用 | 低 |
| 压缩 prompt 上下文 | 30-40% input/agent | 低 |
| 模型降级 | 60-70% 特定调用成本 | 中 |
| 收敛加速 | 减少 1-3 轮 | 低 |
| 模板精简 | 10-15% orchestrator | 极低 |

**综合预估：在典型的 4-6 轮讨论中，总 token 消耗可降低 40-60%。**

---

**潜在风险**

1. 模型降级可能导致 review 质量下降，尤其对需要深度推理的 Athena 和 Momus 角色
2. 观察模式可能错过应该参与的讨论轮次
3. 提案 diff 传递可能导致角色丢失整体上下文
4. 过度压缩 prompt 可能影响角色理解任务

### 🔮 Cassandra (Detail-oriented reviewer):
**Status: Objection** (raw: "## Status: Objection")

I have reviewed this proposal from three dimensions: **logical completeness**, **edge case handling**, and **hidden risks**. I find several specific issues that need to be addressed.

**1. Optimization 1 "Smart Role Selection" lacks a concrete definition of "areas of concern" for each role**

- **Issue:** The proposal says a role enters observation mode when "no new proposal changes involve its area of concern," but nowhere defines what each role's area of concern actually is. How does the orchestrator programmatically determine whether a proposal diff "involves" architecture vs. security vs. feasibility? These domains overlap heavily.
- **Why it matters:** Without a concrete mapping algorithm, this optimization is unimplementable. The orchestrator would need to make subjective judgments about domain relevance each round, which itself consumes tokens and introduces unreliable behavior.
- **Suggested direction:** Either (a) define explicit keyword/section-based heuristics, or (b) simplify to purely status-based: skip a role only after N consecutive "No objection" rounds regardless of content, with a mandatory wake-up every K rounds.

**2. Optimization 2 "Diff-based proposal delivery" will degrade review quality in practice**

- **Issue:** A diff + summary is not equivalent to the full proposal for an agent that has zero prior context. The agent cannot reason about the interaction between changed and unchanged parts if it only sees a summary of the latter.
- **Why it matters:** This is the single highest-risk optimization. Quality degradation would be invisible -- fewer objections looks like faster convergence but is actually missed issues.
- **Suggested direction:** Instead of diff + summary, consider a selective full-section approach or keep full proposal with inline change markers.

**3. Optimization 3 "Model Downgrade" has no defined rollback trigger for edge cases**

- **Issue:** If Sonnet fails to detect an issue and outputs "No objection," the role stays on Sonnet permanently, creating a blind spot. The upgrade depends on the very detection capability that was removed by the downgrade.
- **Why it matters:** Creates a systematic bias toward false convergence with no self-correction mechanism.
- **Suggested direction:** Add a mandatory periodic Opus "check-in" every 3rd round.

**4. Optimization 4 "Convergence Acceleration" conflates "no proposal change" with "proposal is complete"**

- **Issue:** A single round of unanimous "No objection" with no changes could mean the proposal is good, or all roles happened to miss something. The second round is a confirmation check.
- **Why it matters:** The 2-round convergence requirement is the system's primary quality safeguard.
- **Suggested direction:** The 1-round shortcut should only apply when the proposal has been stable for 2+ rounds already.

**5. The proposal lacks a measurement/validation plan for the claimed token savings**

- **Issue:** No baseline measurement, no definition of "typical discussion," no A/B testing plan.
- **Suggested direction:** Run 3-5 representative discussions with the current system and log exact token counts per role per round.

**Key Points:**
- Smart role selection observation mode lacks a deterministic algorithm for domain relevance detection
- Diff-based proposal delivery is the highest-risk optimization because fresh agents cannot reason about interactions between changed and summarized sections
- Model downgrade has a circular dependency: Sonnet must detect issues to trigger its own upgrade
- 1-round convergence shortcut weakens the primary quality safeguard
- Entire proposal lacks measurement baseline and success criteria

**Summary:** Three of five optimizations contain specific logical flaws that would silently degrade review quality while appearing to improve efficiency.

### 🦉 Athena (Strategic architecture reviewer):
**Status: Objection** (raw: "## Status: Objection")

I reviewed from four architectural dimensions: **state management complexity**, **system integrity under optimization**, **coupling between optimizations**, and **phasing strategy**.

**1. Optimization 3 (model downgrade) introduces a hidden quality feedback loop that undermines the system's core value proposition**

- **Issue:** A Sonnet-level agent saying "No objection" carries less epistemic weight than an Opus-level agent. This cheaper confirmation counts toward the 2-round convergence threshold, using a lower-quality signal to make a high-stakes decision (ending debate).
- **Why it matters:** Users will encounter cases where the system converges on flawed proposals because a downgraded role rubber-stamped a round it should have objected to.
- **Suggested direction:** A Sonnet "No objection" should be treated as "Skip" for convergence purposes. Alternatively, restrict downgrade to only Hephaestus and Cassandra, never Athena or Momus.

**2. Optimization 1 (observation mode) and Optimization 2 (diff-only proposal) interact dangerously**

- **Issue:** A role enters observation mode in Round 3, misses Rounds 4-5, then is reawakened in Round 6 with only the Round 6 diff. This role has no knowledge of Rounds 4-5 and no full proposal.
- **Why it matters:** Will produce incoherent reviews immediately upon encountering a multi-round observation gap.
- **Suggested direction:** Any role exiting observation mode MUST receive the full current proposal in its reawakening round.

**Dimensions reviewed without objection:** Phasing strategy (A/B/C) is well-structured. Convergence acceleration (Opt 4) is sound.

**Key Points:**
- Model downgrade creates a perverse incentive: cheaper confirmation signals accelerate convergence on potentially flawed proposals
- Observation mode + diff-only delivery can strand a reawakened role without sufficient context
- Phasing strategy (A→B→C) is architecturally sound
- Convergence acceleration (Opt 4) is clean with low risk
- Removing "Previous Round Full Output" is reasonable given Key Points + Summary mechanism

**Summary:** Optimization 3 threatens convergence integrity, and the interaction between Optimizations 1 and 2 creates a context gap for reawakened roles.

### 🎭 Momus (Devil's advocate):
**Status: Objection** (raw: "## Status: Objection")

**1. "Remove Previous Round Full Output" contradicts how stateless agents maintain reasoning continuity**

- **Issue:** Key Points (3-5 bullets) + Summary (1 sentence) cannot preserve numbered issue lists, reasoning chains, or evidence. Momus raising 4 objections in Round 2 would need to track their resolution in Round 3 — impossible with only Key Points.
- **Why it matters:** Causes either re-raising resolved issues (wasting rounds) or silently dropping valid unresolved objections.
- **Suggested direction:** Implement a structured "previous state" preserving the numbered objection list with accept/reject status, or keep full output with a character limit (~2000 chars).

**2. "Model Downgrade to Sonnet" fundamentally misunderstands what "No objection" means**

- **Issue:** Confirming a complex proposal has no remaining issues requires the same depth of reasoning as finding issues. A weaker model saying "No objection" is not equivalent to a stronger model saying it.
- **Why it matters:** Sonnet fails to notice reintroduced vulnerabilities → system converges with false consensus. Worst possible failure mode: silently less reliable.
- **Suggested direction:** Only downgrade when proposal diff is trivially small (<50 chars). Needs A/B testing with adversarial test cases.

**3. "Observation Mode" has a circular dependency in its wake-up condition**

- **Issue:** The orchestrator (the entity being reviewed) decides whether the reviewer participates. No mechanism for a sleeping role to challenge its own exclusion.
- **Suggested direction:** Remove observation mode entirely, or implement a "headline broadcast" (~100 tokens) letting sleeping roles self-select whether to wake up.

**4. The claimed savings percentages are multiplicatively incompatible**

- **Issue:** If a role is in observation mode, prompt compression and model downgrade on that same role are redundant. Presented as additive when partially overlapping.
- **Suggested direction:** Provide two concrete worked examples with actual token counts.

**5. "Proposal Diff" assumes the orchestrator can produce reliable diffs, but no diff mechanism exists**

- **Issue:** Computing meaningful semantic diffs is a significant new capability being hand-waved as simple.
- **Suggested direction:** Keep full proposal with inline `[CHANGED: ...]` markers instead.

**Key Points:**
- Removing full previous-round output breaks role continuity for tracking objections across rounds
- Model downgrade triggered by "No objection" is dangerous — confirming absence of issues requires same reasoning depth
- Observation mode's wake-up logic has a conflict of interest
- The 40-60% combined savings double-counts overlapping optimizations
- Proposal diff delivery requires significant new capability with correlated failure risk

**Summary:** Two safe optimizations (template trimming, convergence acceleration) but the three major ones each carry risk of silently degrading review quality.

### ⚒️ Hephaestus (Pragmatic execution reviewer):

**1. Optimization 1 lacks concrete mechanism for "relevance detection" and will be expensive to build correctly**

**3. Optimization 3 lacks rollback trigger and quality baseline**

- **Issue:** Sonnet missing issues means it never triggers the upgrade condition. One-way ratchet toward lower quality with no self-correction.
- **Suggested direction:** Add mandatory Opus "spot checks" every 3rd round. Or defer to "future exploration."

**Key Points:**
- Diff-based delivery to stateless agents is structurally broken — drop it, keep the two viable sub-optimizations
- Smart role selection needs a mechanical rule (zero text diff), not LLM-based judgment
- Model downgrade creates a silent failure ratchet — defer or add periodic Opus spot-checks
